# 对外交接文档风格规范

## 适用场景

- 把脚本、实验、工具或项目交给别人使用/复现。
- 写给别人看的数据报告、实验报告或总结文档。
- 任何“不是写给自己看，而是要给另一个独立的人看懂”的文档。

## 核心原则

> **每条信息都必须站在接收者视角。**
> 写之前问自己：接收者看到这句话，能不能立刻知道他要做什么？如果答案是“只有我自己/AI知道”，就删掉或重写。

## 规则

### 1. 可复现脚本/实验文档必须包含

| 项目 | 要求 |
|---|---|
| 最小可复现命令 | 给出完整、可直接运行的命令，不要假设接收者知道前置步骤。 |
| 核心参数解释 | 用自然语言解释每个关键参数影响什么，而不是只列参数名。 |
| 如何改配置 | 明确说明“如果你想改 X，应该改哪里”。 |
| 输出说明 | 说明会生成哪些文件/结果，每个结果代表什么。 |
| 依赖/前提 | 列出接收者必须事先具备的条件。 |

### 2. 数据报告必须包含

| 项目 | 要求 |
|---|---|
| 指标先解释 | 任何指标出现在报告里之前，必须先解释它“是什么、为什么看、单位是什么”。 |
| 自然语言结论 | 不要只贴表格和数字，要有一句人话结论。 |
| 避免自造词 | 不要自己发明或归纳短小词汇/缩写来“总结”信息；如果必须使用，先定义并告知用户。 |
| 上下文 | 解释数据来源、计算方式、实验条件。 |

### 3. 严禁事项

1. **严禁写入只有作者/AI知道的内部信息**
   - 例：内部路径、内部 skill 名称、开发过程、AI 提示词、个人工作习惯。
2. **严禁自造短小词汇或缩写**
   - 例：把“fallback ratio”缩成“FR”、把“end minus start”缩成“E-S”。
   - 如果必须引入新词或缩写，必须在文档中定义，并告知用户。
3. **严禁直接贴出未解释的指标**
   - 任何数字、比率、表格、图表，前面必须有解释。
4. **严禁假设接收者知道背景**
   - 不要写“和之前一样”、“默认那个配置”、“你懂的”。
5. **严禁把 AI 内部视角带进来**
   - 不要写“作为 AI 我建议...”、“本 AI 发现...”。

## 自检清单（写完后必须检查）

- [ ] 接收者是否能不看其他文档就知道这是什么？
- [ ] 是否有最小可复现命令？命令是否完整？
- [ ] 每个核心参数是否用自然语言解释过？
- [ ] 如果想改配置，接收者是否知道改哪里？
- [ ] 每个指标是否在出现前解释过含义和单位？
- [ ] 是否有任何只有作者/AI能看懂的句子？
- [ ] 是否有自造词或缩写没有定义？
- [ ] 输出/结果是否都解释了含义？

## 强制流程

1. 先写初稿。
2. 对照本 SKILL.md 逐条自检。
3.  violations 必须修改，不允许保留。
4. 完成时明确说明“已通过自检清单”。
